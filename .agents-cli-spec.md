# Agent Spec

## Overview
Multi-agent financial coach. Given a user's monthly income, dependants, expenses (manual entry or transaction data via MCP), and debts, produces a budget analysis (categorized spending + recommendations), a savings strategy (emergency fund sizing + savings allocations), and a debt reduction plan (avalanche vs. snowball payoff comparison). Built as an `agents-cli` project for the Kaggle × Google 5-Day AI Agents Intensive capstone.

## Example Use Cases
- Input: `monthly_income=5000`, `dependants=2`, `manual_expenses={Housing:1500, Food:600, ...}`, `debts=[{name:"Credit Card", amount:4000, interest_rate:22, min_payment:100}]` → budget analysis with categorized spending %, 3-5 recommendations with estimated savings; savings strategy with emergency fund target and allocation recommendations; debt plan with avalanche/snowball comparison.
- Input via MCP: user asks the agent to pull recent transactions from the stub transactions MCP server instead of typing expenses manually.

## Tools Required
- Stub MCP server exposing sample/canned transaction data — tool `get_transactions(user_id)` → list of `{date, category, amount}`. No real bank integration, no auth for MVP.

## Constraints & Safety Rules
- Never fabricate a financial figure not derivable from the user's input or transaction data.
- No specific stock/fund/asset picks — vehicle/category-level investing suggestions only (e.g. "a
  low-cost index fund"), and only from `debt-reduction`, gated by its invest-vs-payoff threshold logic.
- Every recommendation must cite the number(s) it's based on.
- Spending category percentages must sum to 100% of `total_expenses`; savings category percentages
  must sum to 100% of `total_surplus`.
- Never recommend reducing or skipping a minimum/regular debt payment, for any reason.
- No persistent storage of income/debt/personal data — in-memory session only for MVP.

## Success Criteria
- Budget categorization: percentages sum to 100%, no uncategorized expense dropped.
- Savings strategy: emergency fund math is correct and traceable to a stated multiplier of expenses.
- Debt plan: avalanche (highest interest first) vs. snowball (smallest balance first) math matches hand-computed ground truth on at least 3 debt scenarios.
- Safety: no investment/buy-sell language in any output; no untraceable numbers.
- Runs end-to-end via `agents-cli playground`, both manual-entry and MCP-transaction paths.

## Architecture & Sub-Agents
- `FinanceCoachWorkflow` (`Workflow` — v2, feature-complete through Phase 3 plus the security
  checkpoint / document upload addition; see `next_steps.md`)
  - `START` → `TransactionFetcherAgent` (`Agent`, no `output_schema` — skill: `transaction-fetcher`
    — normalizes typed text, MCP data, or uploaded PDF statement documents into one JSON picture;
    `notes` field carries any free-text context that isn't income/expenses/debts)
  - → `security_checkpoint` (`@node(rerun_on_resume=True)`, deterministic, no LLM) — always scrubs
    PII and strips injection phrases; routes `"clean"` straight through or raises a `RequestInput`
    confirm-or-stop interrupt when injection is found (`"halted"` route ends the run). See
    `threat_model.md`'s Tampering row for a document-vs-typed-input nuance in PII redaction
    reliability.
  - → `intake_loop` (`@node(rerun_on_resume=True)`) — bounded (2-round) clarification loop; calls
    `IntakeAgent` (`Agent`, `output_schema`) via `ctx.run_node()` — skill: `intake-clarification` —
    decides whether to ask, never analyzes. Pauses via `RequestInput`/resumes via a
    `function_response` addressed to the same `interrupt_id` when clarification is needed.
  - → `analysis_pipeline` (`SequentialAgent` — drops into the `Workflow` edge unchanged)
    - `BudgetAnalysisAgent` (`Agent`, `output_schema`) — skill: `budget-analysis` — descriptive only
    - `SavingsStrategyAgent` (`Agent`, `output_schema`) — skill: `savings-strategy` — prescribes
      spending/savings actions (each tagged `type`: `spending_cut` or `allocation`, for the
      reconciliation identity), analyzes (never prescribes) debt via `debt_context`
    - `DebtReductionAgent` (`Agent`, `output_schema`) — skill: `debt-reduction` — owns all debt and
      invest-vs-payoff prescriptions
    - `OverallPictureAgent` (`Agent`, `output_schema`) — skill: `overall-picture` — merges the above
      into one prioritized, affirming list; adds no new analysis
  - → `critique_refine_loop` (`LoopAgent`, `max_iterations=3` — drops into the `Workflow` edge
    unchanged) — re-checks the whole bundle and fixes what it finds:
    - `CriticAgent` (`Agent`, `output_schema=CriticVerdict`) — skill: `critic` — independently
      re-derives every percentage/reconciliation number, checks realism ceilings, debt-minimum
      safety, and tone; never fixes anything itself
    - `EscalationChecker` (plain `BaseAgent`, no LLM) — stops the loop via
      `EventActions(escalate=True)` once `critic_verdict.approved` is true
    - `RefinerAgent` (`Agent`, `output_schema=RefinedBundle`; runs only when not approved) — skill:
      `refiner` — applies each flagged fix precisely; copies every untouched field forward verbatim
    - `BundleUnpacker` (plain `BaseAgent`, no LLM) — redistributes the refined bundle back into
      `state['budget_analysis']` etc. so the next iteration's `CriticAgent` re-check sees it
- Frontend: `frontend/main.py` — local FastAPI text-entry form, runs the agent in-process via
  `Runner(app=app, ...)`; persists `session_id` across requests and renders/resumes the intake
  loop's clarifying questions via a `function_response` form post.
- **Known gap**: `agents-cli eval generate` doesn't yet support a `Workflow`-typed root agent (see
  `AGENTS.md` → "Known eval-tooling gap"). Verify behavior via direct `Runner` smoke tests instead.

## Data Sources & Auth
- Stub MCP server, sample transaction data, no auth (local/dev only).

## Non-Functional Requirements
- Deployment: prototype first (`--prototype`), then `agent_runtime` via `scaffold enhance` once evals pass.
- Guidance file: `AGENTS.md`.
- No CI/CD for MVP.
